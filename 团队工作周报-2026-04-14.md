### 本周新增 RAG 技术与效果

### 1. Multi-Query 多角度查询扩展

- **技术说明**：对同一个问题生成多个不同表达方式的查询，再把这些查询一起用于检索，尽量减少“用户换一种说法就找不到答案”的情况。
- **参考文献**：[DMQR-RAG: Diverse Multi-Query Rewriting for RAG](https://arxiv.org/abs/2411.13154)（2024）。
- **本周效果**：
  - 在官方知识库 `crud_rag_official_split_local` 的 `RAGAS 30 条` 评测中，`llm_context_recall = 0.9667`，说明当前系统大多数时候已经能找到答题所需的相关材料。
  - 当前结论是“材料基本能找到”，后续主要瓶颈转向“如何把材料组织好并生成更准确的答案”。

### 2. HyDE 假设文档检索

- **技术说明**：先让模型根据问题生成一段“可能的答案描述”，再利用这段描述去帮助检索，从而提升对简短问题或术语不完整问题的理解能力。
- **参考文献**：[Precise Zero-Shot Dense Retrieval without Relevance Labels](https://arxiv.org/abs/2212.10496)（HyDE，2022）。
- **本周效果**：
  - HyDE 已完成接入，主要用于补充语义检索能力，不影响原有关键词检索逻辑。
  - 当前 HyDE 主要承担“补充召回”的作用。从本周评测结果看，系统的材料召回能力已经较强，后续是否默认开启，将结合整体效果继续评估。

### 3. BM25 持久化 + 混合检索

- **技术说明**：将关键词检索所需的索引在建库阶段提前保存好，检索时直接读取；再把“关键词检索”和“语义检索”的结果综合起来，提升检索稳定性。
- **参考文献**：[The Probabilistic Relevance Framework: BM25 and Beyond](https://doi.org/10.1561/1500000019)（2009）。
- **本周效果**：
  - 已完成 `rank_bm25` 持久化接入，知识库重建后可直接加载 `bm25_index.json`。
  - 检索层稳定性增强，配合 dense 检索可保证官方 split 小批量评测中 `retrieval_non_empty = 1.0`。
  - 当前 BM25 已从“性能瓶颈”变成“基础支撑能力”，后续主要优化点转向结果排序和答案生成。

### 4. Corrective RAG 二次检索 + 网络补充搜索

- **技术说明**：在首轮检索后先判断证据是否足够；如果不够，再自动补做一次检索，必要时增加网络搜索结果，减少“信息不全就直接作答”的情况。
- **参考文献**：[Corrective Retrieval Augmented Generation](https://arxiv.org/abs/2401.15884)（2024）；[Self-RAG: Learning to Retrieve, Generate, and Critique through Self-Reflection](https://arxiv.org/abs/2310.11511)（2023）。
- **本周效果**：
  - 已完成 local / temp 知识库双路径接入。
  - 当前版本已具备“本地二次检索 + 部分场景下网络补充搜索”的完整能力。
  - 该技术主要用于降低“只找到部分信息就开始回答”的情况，为后续减少遗漏和补足缺项提供支持。

### 5. 同题文档优先排序，减少跨样本干扰

- **技术说明**：针对 CRUD_RAG 中大量 `news1.txt / news2.txt / news3.txt` 重名文件问题，在结果排序阶段优先保留同一题目下的文档，并降低其他题目中相似但无关内容的权重，减少不同样本内容混入最终引用。
- **实际效果**：
  - 在单样本排查中，最终引用数从 `5` 条收敛到 `3` 条，且都集中在同一题目对应的文档组中。
  - `context_char_f1` 从 `0.1859` 提升到 `0.2793`。
  - `answer_char_f1` 提升到 `0.6537`，`answer_rouge_l_f1` 提升到 `0.5696`。
  - 在 `crud_rag_3qa_60` 全量对比历史基线时：
    - `context_char_f1` 从 `0.1404` 提升到 `0.2173`，提升 `+0.0769`
    - `context_rouge_l_f1` 从 `0.1036` 提升到 `0.1583`，提升 `+0.0547`
    - `reference_count` 从 `5.0` 降到 `3.0`
  - 说明该技术已经明显减少了无关材料混入的问题，检索结果更加集中。

### 技术效果总结

1. **材料召回层面**：Multi-Query、HyDE、BM25 持久化和 Corrective RAG 已经显著降低了“找不到相关材料”的问题，当前 `llm_context_recall = 0.9667`。
2. **结果排序层面**：同题文档优先排序已经有效减少跨样本噪声，是目前检索质量提升最明显的一项。
3. **答案生成层面**：尽管检索和排序已有提升，但当前 `factual_correctness = 0.4670`，说明下一阶段最需要继续优化的是答案生成方式，而不是继续单纯增加召回数量。

### 数据结论

1. 当前系统检索能力较强，`retrieval_non_empty = 1.0` 且 `llm_context_recall = 0.9667`，说明大多数问题已经能找到相关材料。
2. 当前最贴近标准答案的关键片段占比还不够高，`context_char_f1` 和 `context_rouge_l_f1` 仍然偏低。
3. 当前答案生成质量已达到“可以使用、但仍需继续打磨”的阶段，`answer_char_f1 ≈ 0.34`，`faithfulness ≈ 0.72`。
4. 当前最主要短板是 `factual_correctness = 0.4670`，说明答案与标准答案在事实一致性上仍有明显提升空间。

## 三、当前问题

1. 虽然系统大多数时候已经能够找到相关材料，但最贴近标准答案的关键证据比例仍然不足，说明检索结果中仍存在重复和冗余。
2. 当前模型多数情况下会参考证据作答，但答案与标准答案的一致性仍需加强，说明生成环节还有较大优化空间。
3. 评测中仍存在少量失败样本与重复证据问题，若直接扩大到更大规模评测，可能继续影响整体均值。
4. 若后续需要调整 chunk 参数或更换 embedding 模型，将涉及知识库重建，验证时间成本较高。
