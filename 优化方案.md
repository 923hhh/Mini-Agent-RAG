## 一、必须做（阻塞性问题）

### M1 — API Key 泄露（立即处理）

`configs/model_settings.yaml` 中有两个真实密钥已提交到 git：
- DeepSeek key：`sk-***`
- 火山引擎 key：`***`

**方案：** 立即在对应平台轮换这两个 key，然后把 `model_settings.yaml` 改为从环境变量读取，提供 `model_settings.yaml.example` 模板。

### M2 — Embedding 只支持 Ollama，不支持 openai_compatible

`embedding_service.py` 硬编码返回 `OllamaEmbeddings`，但当前配置 `LLM_PROVIDER=openai_compatible`。用户如果没有本地 Ollama，知识库构建和检索全部失败。

**方案：** 仿照 `llm_service.py` 的 provider 分支，为 `embedding_service.py` 增加 `openai_compatible` 路径（支持 OpenAI 兼容的 embedding 接口，如 DeepSeek、SiliconFlow 等）。

### M3 — `load_settings()` 每次请求都重读 3 个 YAML

每个 API 请求都触发 3 次磁盘 I/O，高并发下是明显瓶颈。

**方案：** 用 FastAPI `lifespan` 在启动时加载一次，通过依赖注入传递，或加 `@lru_cache`。

## 二、可选优化（质量提升）

### O1 — `documents.py` 拆分（2790 行单文件）

当前一个文件承担：格式解析、OCR 流水线、VLM 调用、图像预处理、指令页检测、元数据提取。

**方案：**
```
app/loaders/
  ├── text.py       # txt/md
  ├── pdf.py        # PDF + 扫描件
  ├── office.py     # docx/epub
  ├── image.py      # 图像 + OCR
  ├── vlm.py        # VLM 描述调用
  └── factory.py    # KnowledgeFactory 分发
```

### O2 — 提取重复代码

| 重复项 | 出现位置 | 提取目标 |
|--------|----------|----------|
| `_coerce_optional_text()` | `kb_incremental_rebuild.py` + `embedding_assembler.py` | `app/utils/text.py` |
| `IMAGE_QUERY_HINTS` | `rag.py` + `local_kb.py` | `app/constants.py` |
| header 提取逻辑 | `embedding_assembler.py` + `local_kb.py` | `app/utils/text.py` |
| 去重函数 | 3 处不同实现 | 统一为一个 |

### O3 — 知识库重建接口异步化

`POST /knowledge_base/rebuild` 是同步阻塞的，大型知识库会超时（已知问题，TODO 文档有记录）。

**方案：** 改为异步任务模式：
- `POST /knowledge_base/rebuild` 返回 `{task_id, status}`
- `GET /knowledge_base/rebuild/{task_id}` 查询进度和结果

### O4 — 删除 `agents/executor.py`

该文件只有两行 re-export，没有实际价值，增加导入路径混乱。直接删除，调用方改为从 `multistep.py` 导入。

### O5 — Tesseract 路径配置化

`kb_settings.yaml` 中硬编码了开发者本机路径 `C:/Users/ASUS/tesseract-local/tesseract.exe`，其他人克隆后直接失败。

**方案：** 改为相对路径 `./data/tools/Tesseract-OCR/tesseract.exe`，或支持环境变量覆盖。

### O6 — Agent 工具选择升级（Phase L 前置）

当前基于关键词列表匹配，语义模糊时不稳定。

**方案：** 接入 LLM 原生 tool calling（DeepSeek 已支持），替换关键词匹配逻辑。

## 六、xMemory 映射到当前项目的改造方案

### 6.1 摘要

目标是把 xMemory 映射成**新增的 Agent 长期记忆分支**，而不是重写当前通用文档 RAG。第一版采用**独立 memory store**，范围只覆盖 `/chat/agent`，不开放独立 memory API，不改现有 `/chat/rag` 主链路。

第一版只做到 `raw turns -> episodes -> semantic facts` 三层，不做 `theme` 层。核心收益是：把当前只靠 `history` 的短期上下文，升级为可跨请求复用、可去重、可按语义聚合的长期记忆，同时保持现有知识库检索和多模态能力不受影响。

### 6.2 关键改造

#### 6.2.1 新增独立 Agent Memory 子系统

- 新建独立存储根目录，采用 `data/agent_memory/<session_id>/` 结构，不与现有知识库目录混用。
- 每个 `session_id` 下保存三类数据：
  - `turns.jsonl`：原始会话 turn，追加写入，作为最底层 faithful memory。
  - `episodes.jsonl`：由连续 turn 聚合出的 episode 记录，包含 `episode_id`、turn 范围、时间戳、标题、摘要、参与角色、状态。
  - `semantics.jsonl`：从 episode 提炼出的长期事实，包含 `semantic_id`、事实文本、来源 `episode_id`、事实类别、置信度、失效/更新时间。
- 为 `episodes` 和 `semantics` 分别建立独立向量索引；`turns` 不直接做主检索，只作为低层展开证据。

#### 6.2.2 第一版 memory construction 策略

- `turns -> episodes`：优先采用启发式分段，不在 v1 依赖复杂 LLM 边界检测。
- 分段信号固定为：时间间隔、显式换题、工具调用阶段结束、用户任务切换、知识库切换。
- 默认策略：达到一定 turn 数或触发明显边界时封存一个 episode。
- `episodes -> semantics`：用 LLM 对 episode 做“长期有价值事实”抽取。
- 只抽取持久、可检索、可独立理解的信息，如偏好、约束、目标、身份、项目状态、关键决策。
- 不抽取短暂寒暄、一次性情绪、无后续价值的回复。
- 提取失败时保留 episode，不阻断主流程。
- v1 不做 `theme` 层，也不做复杂 merge/split；但 episode 和 semantic 记录都预留 `parent/children/source_ids` 字段，方便二阶段升级。

#### 6.2.3 memory retrieval 按 xMemory 思路做 top-down

- Agent 请求进入后，先做 memory retrieval，再做工具规划和知识库检索。
- memory retrieval 采用两阶段：
  - Stage A：先检索 `semantic` 层，取相关且去重后的事实集合，作为高层代表节点。
  - Stage B：根据命中的 `semantic_id -> episode_id` 反查 episode，只补入能增加新信息的 episode 摘要；原始 turn 只在证据不足或存在冲突时再展开。
- 上下文扩展遵循“先抽象、后展开”的固定策略：
  - 默认只注入 `semantic + episode summary`
  - 若多条 semantic 冲突、semantic 过短、或问题包含明显时间/因果依赖，再补原始 turn 片段
- 明确不做通用 RAG 的固定 top-k 拼接；memory 侧目标是**更少 token、更多非重复证据**。

#### 6.2.4 Agent 接入方式

- `/chat/agent` 新增可选 `session_id`，作为长期记忆主键。
- v1 不新增公开的 memory tool；memory retrieval 作为 Agent 执行前置步骤自动运行。
- Agent 规划 prompt 增加 memory context 段，优先用长期记忆回答“你之前说过/上次我们讨论过/我偏好什么/当前项目进行到哪”的问题。
- 最终回答阶段把 memory context 与知识库 references 分开组织：
  - `长期记忆证据`
  - `知识库证据`
- 若 memory 与知识库冲突，优先提示冲突，不自动融合。

#### 6.2.5 配置与可观测性

- 新增 Agent memory 配置，默认关闭：
  - `ENABLE_AGENT_MEMORY=false`
  - `AGENT_MEMORY_MAX_RECENT_TURNS`
  - `AGENT_MEMORY_EPISODE_MAX_TURNS`
  - `AGENT_MEMORY_SEMANTIC_TOP_K`
  - `AGENT_MEMORY_EPISODE_TOP_K`
  - `AGENT_MEMORY_ENABLE_TURN_EXPANSION`
- 复用现有 trace 体系，新增：
  - `memory_build_trace.jsonl`
  - `memory_retrieval_trace.jsonl`
- trace 至少记录：
  - 当前 `session_id`
  - 新增 episode 数
  - 新增 semantic 数
  - 命中 semantic / episode / turn 数
  - 最终 memory token 预算占用
- v1 不做 UI 页面；只要求后端 trace 可用于调试。

### 6.3 对外接口与类型变化

- `AgentChatRequest`
  - 新增 `session_id: str | None`
  - 未传时默认无长期记忆，只走当前行为，保证兼容
- `AgentChatResponse` 与流式 `done` 事件
  - 回传 `session_id`
  - 可选回传 `memory_overview`，至少包含 `semantic_hits`、`episode_hits`、`turn_hits`、`used_memory`
- 内部新增 memory record 类型
  - `MemoryTurn`
  - `MemoryEpisode`
  - `MemorySemantic`
  - `MemoryOverview`

### 6.4 分阶段实施

#### Phase 1：最小可落地版

- 加 `session_id`
- 建独立 memory store
- 记录 raw turns
- 按启发式生成 episode
- 从 episode 提取 semantic
- Agent 自动检索 semantic + episode 摘要
- 只做 trace，不做 UI

#### Phase 2：xMemory 风格检索增强

- semantic 检索增加去重与覆盖优先，而不是纯相似度 top-k
- episode inclusion 改成“增益式补充”，避免重复 episode 一起进入上下文
- 冲突语义触发原始 turn 展开
- 在最终回答中区分 memory 证据与 KB 证据

#### Phase 3：高级增强

- 引入 `theme` 层
- 支持 semantic reassignment / merge / split
- 增加 memory-only 评测集
- 再考虑调试 API 或 UI

### 6.5 测试计划

- 兼容性
  - 不传 `session_id` 时，`/chat/agent` 行为与当前一致
  - `/chat/rag` 完全不受影响
- 记忆写入
  - 同一 `session_id` 连续多次请求后，turn/episode/semantic 正常累积
  - 更换 `session_id` 不串记忆
- 检索正确性
  - 用户在前几轮提到偏好、约束、项目状态，后几轮能被 semantic 命中
  - 涉及时间链的问题，能从 semantic 回溯到 episode，必要时再展开 turn
  - 重复表达不会导致 memory context 大量冗余
- 冲突处理
  - 同一事实被更新时，新旧 semantic 不应同时被无差别引用
  - memory 与 KB 冲突时，答案要明确指出冲突来源
- 质量与效率
  - 相比直接拼接全部 history，memory context token 数明显下降
  - 在跨请求问题上，答案质量优于仅靠 `history` 的版本

### 6.6 假设与默认值

- 这版方案只针对 Agent 长期记忆，不推广到通用文档知识库主链路。
- memory store 独立存储，但向量索引和 embedding 设施可复用现有实现。
- v1 采用启发式 episode 分段，避免一开始把边界检测完全交给 LLM。
- v1 的 semantic 抽取允许失败回退，不要求严格 schema 才继续主流程。
- v1 不做 theme、retroactive merge/split、独立 memory API、UI 调试页。
