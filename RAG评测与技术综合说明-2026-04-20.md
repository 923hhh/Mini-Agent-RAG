- `Recall@5`：前 5 条结果里有没有找到正确证据。

- `NDCG@5`：前 5 条结果的整体排序质量好不好。

#### 1.5 结果解读

##### 1.5.1 CRUD 结果如何理解

- `CRUD` 的三项指标都是 `1.0`，说明在当前项目内知识库和评测集强对齐的前提下，系统检索表现非常稳定。
- 但这更适合解释为“项目内闭环检索很稳定”，不建议直接表述成“开放场景检索已经完美”。
- 换句话说，这个结果说明基础链路已经打通，并且在当前知识库组织方式下运行良好。

##### 1.5.2 Domain 结果如何理解

- 从初始基线到当前稳定版，`Domain` 的三项检索指标都已经明显提升。
- `Recall@5 = 0.8900` 说明大多数问题在前 5 条里已经能找到正确证据。
- `MRR = 0.7340` 和 `NDCG@5 = 0.7379` 说明当前已经不只是“能找到”，而且排序质量比最初稳定了很多。
- 但这并不代表问题已经完全解决，离当前阶段目标 `MRR ≥ 0.75`、`NDCG ≥ 0.75` 仍然还差一点。

##### 1.5.3 分任务结果说明了什么

- `structured_qa` 仍然最好，说明在结构化、边界清晰的问题上，系统已经比较成熟。
- `conversation_qa` 当前也比较稳，说明多轮上下文利用不是当前主矛盾。
- `multi-doc_qa` 的排序质量已经比初版明显改善，说明“多源覆盖优化”是有效的。
- `time-sensitive_qa` 依然最弱，但已经从最初的 `0.4000 / 0.2050 / 0.2511` 提升到 `0.6000 / 0.3183 / 0.3874`，说明时间约束保留、日期元数据和后续 candidate 扩深确实带来了正向收益。
- `extractive_qa` 仍然是后续重点，因为它直接影响 `Top1`、`MRR` 和生成阶段的事实正确率。

#### 1.6 小结

- 当前 `CRUD` 检索结果主要说明：系统在项目内闭环场景下已经比较稳定。
- 当前 `Domain` 检索结果主要说明：系统已经从“可用但偏弱”推进到“整体可用且排序明显改善”的阶段。
- 现在最值得继续推进的，不再是单纯提高召回，而是：
  - 继续提升 `extractive_qa` 的 `Top1 / MRR`
  - 继续拉升 `time-sensitive_qa`


### 2. 生成评测结果（RAGAS）

本轮评测目标是在约 1 小时内完成生成侧质量对比，因此采用了缩减样本，而不是大规模全量评测。

补充说明：

- 截至 `2026-04-20`，生成侧最近几轮主要是在做小步实验和保留策略筛选。
- 当前文档里仍然保留的是 `2026-04-17` 这轮 `10 + 10` 的 `RAGAS` 缩减样本结果。


#### 2.1 评测口径

- CRUD：`10` 条
- Domain：`10` 条
- Domain 采用均衡抽样：`extractive_qa / conversation_qa / multi-doc_qa / time-sensitive_qa / structured_qa` 各 `2` 条

#### 2.2 指标说明

- `Context Recall`：回答有没有覆盖检索到的关键信息。
- `Faithfulness`：回答是否基本遵循了给定证据，而不是脱离证据自由发挥。
- `Factual Correctness`：回答本身在事实层面是否正确。
- `Answer Relevance`：回答是否真正对准了问题。

需要特别说明的是：

- 这轮只是一版“趋势判断”性质的生成侧评测，不适合直接当作最终 benchmark。
- 样本量较小，因此更适合用来看方向，而不是用来下特别绝对的结论。
- 另外，`RAGAS` 分数高低只代表某一维度的相对表现，不能把单个指标单独理解成系统整体质量。

#### 2.3 结果文件

- CRUD 结果：`data/eval/crud_rag_3qa_full_ragas_10.json`
- Domain 结果：`data/eval/domainrag_ragas_balanced_10_ragas_10.json`
- Domain 均衡评测集：`data/eval/domainrag_ragas_balanced_10_domainrag_small_batch.jsonl`

#### 2.4 总体结果

| 数据集 | 样本数 | Context Recall | Faithfulness | Factual Correctness | Answer Relevance |
| --- | --- | --- | --- | --- | --- |
| CRUD | `10` | `1.0000` | `0.7288` | `0.4540` | `0.4841` |
| Domain | `10` | `0.8333` | `0.5284` | `0.0570` | `0.2402` |

#### 2.5 结果解读

##### 2.5.1 CRUD 结果如何理解

- `CRUD` 的 `Context Recall = 1.0000`，说明在这批样本上，系统通常能把检索到的关键信息带进回答里。
- 但 `Factual Correctness = 0.4540` 并不高，这说明“用到了上下文”并不等于“回答就一定完整且正确”。
- 因此，当前 `CRUD` 更适合说明系统基础链路已经可用，而不能简单理解成生成质量已经很好。

##### 2.5.2 Domain 结果如何理解

- `Domain` 的四项指标整体都低于 `CRUD`，说明在更复杂、更接近真实场景的问题上，生成质量下降更明显。
- 其中 `Faithfulness` 和 `Answer Relevance` 偏低，说明当前回答更容易出现两类问题：
  - 没有完全贴住证据
  - 没有完全对准问题
- `Factual Correctness = 0.0570` 是最弱的一项，这说明当前最突出的问题已经不是“能不能生成回答”，而是“生成出来的回答是否真的正确”。

##### 2.5.3 这组结果最值得关注什么

- `Context Recall` 高，并不代表回答就一定正确。
- 当前更关键的矛盾，在于生成阶段的“事实级正确性”仍然偏弱。
- 换句话说，现在系统更需要解决的是“证据已经拿到了，但答案仍然不够准确、不够完整”的问题。

#### 2.6 小结

- 在这轮缩减样本评测里，`CRUD` 的生成侧表现明显好于 `Domain`。
- 当前最值得优先关注的生成侧问题是 `Factual Correctness`。
- 如果后续继续优化，生成侧的重点不应只是“让答案更长”或“让答案更流畅”，而应优先提高“是否真正基于证据、是否事实正确”。


### 3. 评测结果总判断

把检索结果和生成结果合在一起看，可以得到一个更完整的判断：

- 当前系统在 `CRUD` 数据集上已经比较稳定，说明基础链路是通的。
- 当前系统在 `Domain` 数据集上，检索和生成两侧都暴露出更真实的困难场景。
- 当前最值得优先优化的，不是单纯继续增加功能，而是把 `Domain` 场景下的排序精度和事实正确性做上去。

## 四、技术实现

### 1. 文档说明

这一部分围绕当前项目里真正启用的 RAG 主链路整理，重点展示三件事：

- 当前系统到底用了哪些 RAG 技术
- 这些技术在代码里分别在哪里
- 每一部分最关键的代码片段是什么

需要说明的是：

- 本部分优先写当前主链路里真正启用、真正参与评测的部分。
- 对于“代码里存在，但当前默认未启用”的路径，也会明确标注，避免把“有这段代码”误解成“当前就在运行”。
- 每个技术点都分成三部分：
  - 通俗解释
  - 代码位置
  - 核心代码片段
- 代码片段不追求“整段全贴”，而是只保留最能体现逻辑的核心部分。

### 2. 当前 RAG 主链路结论

当前系统不是“单次向量检索 + 直接拼 prompt”的基础版 RAG，而是下面这套增强链路：

1. 查询改写与多查询检索
2. 向量检索 + BM25 混合检索
3. 启发式重排 + 模型重排
4. 答案句窗口增强与句级索引提示
5. `small-to-big` 上下文扩展
6. 文本 / OCR / 视觉证据分组组织
7. 生成后完备性补查 + 事实审校

可以把当前主链路概括成：

`用户问题`
→ `查询改写 / 多查询`
→ `dense + BM25 混合检索`
→ `启发式重排`
→ `模型重排`
→ `答案句窗口 / 句级提示`
→ `small-to-big 扩展上下文`
→ `按文本 / OCR / 视觉组织证据`
→ `生成答案`
→ `完备性补查`
→ `事实审校`
→ `最终输出`

- 句级索引开着，但当前是保守接入

- “这条路径当前默认真的在运行”

### 4. 核心技术与代码片段

#### 4.1 查询改写与多查询检索

通俗解释：

用户问题有时太口语、太短、太模糊，直接检索不一定好。系统会先判断是否值得改写；如果值得，就优先生成多条检索 query；如果多查询失败，再退回单查询改写。

代码位置：

- `app/services/query_rewrite_service.py:86-118`

核心代码片段：

```python
def generate_multi_queries(
    settings: AppSettings,
    query: str,
    history: list[ChatMessage] | None = None,
) -> list[str]:
    normalized_query = query.strip()
    if not normalized_query:
        return []
    if not settings.kb.ENABLE_QUERY_REWRITE or _should_skip_rewrite(normalized_query):
        return [normalized_query]

    max_queries = settings.kb.MULTI_QUERY_MAX_QUERIES
    if max_queries <= 1:
        return [normalized_query]

    if settings.kb.ENABLE_MULTI_QUERY_RETRIEVAL:
        generated_queries = _invoke_multi_query_rewrite(
            settings=settings,
            query=normalized_query,
            history=history,
            max_queries=max_queries,
        )
        if generated_queries:
            return deduplicate_query_candidates(
                normalized_query,
                generated_queries,
                limit=max_queries,
            )

    rewritten = _invoke_single_query_rewrite(settings, normalized_query, history)
    if not rewritten:
        return [normalized_query]
    return deduplicate_query_candidates(normalized_query, [rewritten], limit=max_queries)
```

这段代码最核心的意思是：

- 太短、太简单的问题直接跳过改写
- 默认最多生成 `3` 条 query
- 优先走多查询
- 多查询失败才退回单查询

也就是说，当前不是“先问什么就搜什么”，而是“先把问题改成更适合检索的形式再搜”。

max_lexical = max((item.lexical_score for item in fused), default=1.0) or 1.0
for item in fused:
    score = 0.0

        score += 1.0 / (settings.kb.HYBRID_RRF_K + item.dense_rank)
    if item.lexical_rank is not None:

- 第一层：启发式重排
- 第二层：模型重排

##### 4.4.1 启发式重排

- 它非常适合当前 `CRUD` 这类项目内闭环数据
- 但不应直接把它理解成开放场景下的通用最优策略

#### 4.5 `small-to-big`：检索时小块命中，回答时扩成大块

通俗解释：

小块更容易检索到，但只给模型一个小块，往往上下文不够。所以当前项目会在最终转 `reference` 时，把命中的 chunk 向前后扩一圈。

代码位置：

- `app/retrievers/local_kb.py:725-800`

核心代码片段：

```python
expanded_content = build_expanded_content(
    settings=settings,
    document=candidate.document,
    grouped_documents=grouped_documents,
)

def build_expanded_content(
    *,
    settings: AppSettings,
    document: Document,
    grouped_documents: dict[str, dict[int, Document]],
) -> str:
    if not settings.kb.ENABLE_SMALL_TO_BIG_CONTEXT:
        return document.page_content

    expand_chunks = settings.kb.SMALL_TO_BIG_EXPAND_CHUNKS
    for index in range(chunk_index - expand_chunks, chunk_index + expand_chunks + 1):
        item = available.get(index)
        if item is None:
            continue
        text = item.page_content.strip()
        if not text or text in seen_texts:
            continue
        pieces.append(text)

    return "\n".join(pieces) if pieces else document.page_content
```

当前配置里 `SMALL_TO_BIG_EXPAND_CHUNKS = 1`，也就是：

- 命中 1 个 chunk
- 回答时把前后各 1 个相邻块一起拼回去

所以当前项目实际上走的是“小块检索，大块回答”。

#### 4.6 多模态证据组织：文本 / OCR / 视觉描述分开拼

通俗解释：

当前系统不是把所有证据混成一坨文本，而是会先判断当前问题是不是偏图片理解，然后再把证据按类型分组组织。

代码位置：

- `app/chains/rag.py:318-342`
- `app/chains/rag.py:370-399`

核心代码片段 1：判断是否走图片型 prompt

```python
def should_use_image_rag_prompt(
    query: str,
    references: list[RetrievedReference],
) -> bool:
    lowered_query = query.strip().lower()
    query_hint_hit = any(hint in lowered_query for hint in IMAGE_QUERY_HINTS)

    image_evidence_count = 0
    text_evidence_count = 0
    for ref in references:
        group = resolve_reference_context_group(ref)
        if group in {"ocr", "vision"}:
            image_evidence_count += 1
        else:
            text_evidence_count += 1

    if query_hint_hit and image_evidence_count > 0:
        return True
    if image_evidence_count > 0 and text_evidence_count == 0:
        return True
    if image_evidence_count >= 2 and image_evidence_count >= text_evidence_count:
        return True
    if query_hint_hit and not references:
        return True
    return False
```

#### 4.7 生成后完备性补查与事实审校

通俗解释：

当前项目不是检索完直接回答就结束，而是：

1. 先基于证据生成初稿
2. 再检查这个初稿有没有漏答
3. 再检查这个答案里有没有证据未明确支持的内容
4. 如果有问题，就让模型基于原证据修订一次

代码位置：

- `app/chains/rag.py:97-122`
- `app/chains/rag.py:583-627`

核心代码片段 1：先生成，再补查

```python
def generate_rag_answer(
    settings: AppSettings,
    query: str,
    references: list[RetrievedReference],
    history: list[ChatMessage],
    agent_memory_context: str = "",
) -> str:
    prompt_kind = resolve_rag_prompt_kind(query, references)
    prompt = build_rag_prompt(query, references)
    variables = build_rag_variables(query, references, history, agent_memory_context=agent_memory_context)
    llm = build_chat_model(settings, temperature=0.0)
    chain = prompt | llm | StrOutputParser()
    answer = chain.invoke(variables)
    answer = maybe_refine_rag_answer(
        settings=settings,
        query=query,
        references=references,
        context=str(variables["context"]),
        coverage_requirements=str(variables["coverage_requirements"]),
        draft_answer=answer,
    )
    return answer
```

这两段说明当前系统已经有一个很实用的后处理逻辑：

- 如果问题本身像“哪些、分别、同时、原因、措施、职责”这类容易漏项的问题
- 就会触发一次完备性审校
- 如果答案里出现证据未明确支持的外推内容，还会再做一次事实审校
- 若发现问题，就直接返回修订版答案

所以当前项目不是“生成完就算了”，而是会再做一次“答全了没有”和“有没有说过头”的检查。

同时也要说明：

- 这一步是轻量的答案审校
- 不是重新联网检索，也不是新一轮外部证据补充
- 因此它更适合解决“漏项”和“轻度外推”问题，不足以单独解决“证据本身没找对”的问题


### 5. 当前代码层面的真实结论

如果只用一句更准确的话概括当前项目，可以这样说：

“当前系统已经不是基础版 RAG，而是一个带查询改写、混合检索、双层重排、答案句窗口增强、句级索引提示、small-to-big、多模态证据组织，以及生成后完备性与事实审校的增强型 RAG；现阶段最主要的问题不是完全检索不到，而是 Domain 场景下 `extractive_qa` 的 Top1、时间敏感排序，以及生成事实正确性还需要继续优化。”
